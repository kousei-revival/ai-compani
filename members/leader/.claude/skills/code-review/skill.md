# code-review

感想ではなく、リスクの高い順にレビューする。

## 優先順位

1. バグや仕様逸脱
2. 回帰リスク
3. 例外処理や境界条件
4. 設計上の無理
5. テスト不足
6. 細かい改善案

## 見る観点

- 入力が崩れたときにどうなるか
- 既存の使い方を壊さないか
- 変更の意図に対して実装が過不足ないか
- 複雑さが不必要に増えていないか
- テストは重要経路を押さえているか

## 出し方

- まず `Findings` を重い順に出す
- 各指摘は `何が問題か / なぜまずいか / 何が起きうるか` を書く
- 問題がなければ、その旨を明言する

## 出力フォーマット

```md
## Findings
- [severity] 問題の要約

## Open Questions
- 必要なら確認事項

## Summary
- 全体所感
```
