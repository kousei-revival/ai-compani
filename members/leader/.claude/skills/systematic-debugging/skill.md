# systematic-debugging

思いつきで直さず、証拠を集めてから直す。

## 鉄則

`根本原因を調べる前に修正しない`

## 4フェーズ

### Phase 1: Root Cause

- エラーメッセージを読む
- 安定再現できるか確認する
- 最近の変更を見る
- 多層構造なら境界ごとに証拠を取る

### Phase 2: Pattern

- 動いている似た例を探す
- 壊れている例と比較する
- どこが違うか列挙する

### Phase 3: Hypothesis

- 仮説を1つだけ立てる
- 最小変更で確かめる
- 外れたら仮説を更新する

### Phase 4: Implementation

- 失敗するテストを作る
- 単一修正を入れる
- 直ったか検証する

## 赤信号

次が出たら止まる。

- とりあえずこれを変えてみる
- 多分これが原因だと思う
- いくつかまとめて直す
- テストはあとで書く

## 出力フォーマット

```md
## Root Cause Investigation
- 証拠1
- 証拠2

## Hypothesis
- 仮説

## Next Test
- 何で確かめるか

## Fix
- 単一修正
```
