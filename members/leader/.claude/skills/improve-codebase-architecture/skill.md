# improve-codebase-architecture

読みづらさや直しづらさを、構造の問題として見る。

## 基本ルール

- いきなり設計案を1つに決めない
- まず `どこに摩擦があるか` を集める
- 浅いモジュールを深くできないか考える
- テストのしやすさも評価軸に入れる

## 進め方

### Step 1: 摩擦を探す

- 概念理解に何ファイルも行き来する
- 境界が薄く、呼び出し側が複雑
- テストが内部構造に寄りすぎている
- 変更が複数箇所へ波及しやすい

### Step 2: 候補を列挙する

各候補について見るもの。

- どの概念が散っているか
- なぜ結合しているか
- 深くできそうか
- テストはどう変わるか

### Step 3: 1つ選んで深掘る

- どんな境界にしたいか
- 何を隠したいか
- どの依存を飲み込ませたいか

### Step 4: 改善案を比べる

- 最小インターフェース案
- よく使うケース最適化案
- 柔軟性重視案

## 出力フォーマット

```md
## Candidate
- どの領域か
- 何がつらいか

## Deepening Direction
- 何を隠すか
- 境界をどうするか

## Expected Impact
- テスト
- 読みやすさ
- 変更容易性
```
