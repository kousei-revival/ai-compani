# 会社名

## 使命

小さな会社でも大企業と戦える仕組みを作る。

## メンバー（役職）の決め方

経営者が、**自分の会社で「週 3 回以上やる作業」**を書き出す。  
その作業単位が、そのまま **AI 社員の役職候補** になる。

- 各部屋は `members/<役職名>/`（例: `members/writer/`）
- 役割はそのフォルダの `CLAUDE.md` に固定する（**雛形:** `templates/member-CLAUDE-template.md` をコピーして埋める）
- 頻度が下がったら兼任や統合を検討する

| 作業の例（週 3 回以上） | 役職の例 |
|-------------------------|----------|
| 問い合わせ対応 | カスタマーサクセス社員 |
| SNS 投稿 | SNS 担当社員 |
| リサーチ | リサーチャー社員 |
| 動画編集 | 動画担当社員 |
| 数値分析 | アナリスト社員 |

## 新役職を作る前に答える5つ

`members/<名前>/` を新設する**前**に、経営者（必要なら leader）が次を埋める。  
**曖昧なまま作ると AI も困る。**ここで解像度を上げてからフォルダと `CLAUDE.md` を切る。

☐ **この役職の目的は？**（何を生む人か）  
☐ **判断基準は何で測る？**（数字・品質・スピード）  
☐ **成果物はどの形式？**（テキスト・画像・数値）  
☐ **誰に報告する？**（leader？経営者？）  
☐ **週に何時間分の仕事量？**

埋まった内容は、そのまま当該 `CLAUDE.md` の **役割・判断基準・成果物ルール・報告先・負荷の目安** に写す。雛形は **`~/ai-company/templates/member-CLAUDE-template.md`**。

**既存役**（`leader` / `writer` / `designer` / `sns` / `academic` / `customer_success` / `operations` / `web` / `video` / `researcher`）の各 `CLAUDE.md` 末尾にも **「要件定義（5つ）」** 表を置いた。数値は目安なので、実態とずれたら経営者が更新する。

## CLAUDE.md の育て方（50 点スタート）

初版の `members/*/CLAUDE.md` は **50 点でよい**。完璧を待たずに運用を始める。

運用しながら、気づきを**その場で同じファイルに追記**する。

- 「この判断ミスは避けたい」→ **判断基準**に追記
- 「この成果物の品質が欲しい」→ **成果物ルール**に追記
- 「このパターンは NG」→ **NG 事例**に追記（セクションが無ければ `## NG 事例` を新設）

**フィードバックを会話で終わらせず、即ファイルに書く習慣**＝AI 社員の育成そのもの。leader は、経営者・自分自身から出た方針が該当 `CLAUDE.md` に反映されたかを、夕方のタイミングで軽く確認してよい。

## CLAUDE.md の月次メンテ（肥大化対策）

`members/*/CLAUDE.md` および **`~/ai-company/CLAUDE.md` 自体**は、放っておくと伸び続ける。**月 1 回**、経営者と leader が見直す。

- **もう守ってないルール** → **削除**する（形骸化した一文は害になる）
- **重複している項目** → **統合**する（同じことを二箇所で言わない）
- **抽象的すぎる箇条書き** → **具体化**するか、測れないなら **削除**する

**短く・具体的・守れるもの**が一番効く。**ルールを増やすより減らす**ことでチームが強くなる。

## 基本ルール

- 日本語で応答
- 一次情報を残す（LOG.md）
- 数字で語る
- 足すな、削れ

## 報告フロー

- leader が全メンバーの動きを把握
- 成果物は leader レビュー → 経営者報告

## AI 営業（4 役連携）

**事業構築セミナー**などで **AI 営業を実演し、同じ手順で実践する**ときの役割分担。専任の `members/sales/` は置かず、次の既存役でパイプラインを回す（詳細は各 `members/<役>/CLAUDE.md` の「営業との連携」）。

| 段階 | 役 | ざっくりやること |
|------|-----|------------------|
| 入口・機会 | `researcher` | 案件・媒体のリサーチ、候補と優先度、応募たたき台の素材（送信は人） |
| 認知・信頼 | `sns` | 実演で見せた価値・学びを、トーンを守った投稿案・ストックにする |
| 契約後 | `customer_success` | ロードマップ・日報・次アクションなど、伴走のたたき台 |
| 監督・配管 | `leader` | dispatch / handoff、レビュー、経営者向けサマリ。**セミナー週のテーマ・実演メモ**を inbox に落として各役へ流す |

### セミナー・実演用 `projects/` の命名（全役で共通）

**1 回のセミナー（または同じカリキュラムのまとまり）につき、フォルダ名を 1 つ決め、** `researcher` / `sns` / `customer_success` では **同じ名前**の `projects/<名前>/` を使う。あとから「あの講義の成果どこ？」で横断検索しやすくするため。

| ルール | 内容 |
|--------|------|
| **基本形** | `YYYY-MM-seminar-<英語の短いスラッグ>/`（スラッグは **小文字・ハイフン区切り**） |
| **例** | `2026-04-seminar-ai-sales` |
| **同月に複数回あるとき** | 開催日で区切る: `2026-04-23-seminar-ai-sales` — または `2026-04-seminar-ai-sales-week2` のように末尾で区別 |
| **leader** | 集約メモ・週次サマリを `members/leader/projects/<上と同じ名前>/` に置いてもよい（任意） |
| **クライアント伴走** | 従来どおり `customer_success` は `projects/<クライアントコード>/` も使う。**セミナー教材・デモ再現**と**契約後の客先フォルダ**は分ける |

**実践のコツ（初心者向け）**

1. 上の **命名を決めたら**、実演のプロンプト・条件・文案は **`members/<役>/projects/<その名前>/`** に Markdown で残す（依頼・dispatch にも同じパスを書く）。
2. 公開に近い文面（投稿・クライアント送付）は必ず **leader レビュー** を挟む（会社の `報告フロー` と同じ）。
3. 「いまどの段階の商談か」を leader が短くラベル化すると、researcher / sns / customer_success の依頼がブレにくい。

## 一日のリズム

- **朝**：leader が全員の inbox を確認 → 当日タスクリスト提出
- **日中**：各メンバーが実務を実行
- **夕方**：成果と明日の予定を leader が集約して経営者に報告
- **手渡しの正本**（ToDo マトリクス・今日 7 件など）: **`meetings/handoffs/YYYY-MM-DD.md`** を `templates/council-handoff-template.md` から毎回生成（手順は `meetings/README.md`）

このリズムを決めると運用がブレない。
